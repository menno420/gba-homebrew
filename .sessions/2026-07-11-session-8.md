## Row #23 — root cause, fix, proof (CLOSED)

**Root cause:** `row_features` places each 2-cell crystal cluster at one edge
of its row's open span using only that row's own layout. At echo-band
junctions where the band width changes, the span of columns OPEN IN BOTH
consecutive rows can shrink to exactly those 2 cells — every crossable column
carries a crystal, an unavoidable death gate. Slice 2's scan (rows ≤1200)
found 6 (junctions 191/192, 287/288, 317/318, 767/768, 967/968, 1093/1094);
over the full generation the true count was **126 gates across two 10080-row
echo periods** (63 per period, repeating forever).

**Fix (pure, neighbour-aware, minimal):** in `row_features`, each crystal
cell is kept only if at least one crystal-free shared-open column survives
toward BOTH neighbouring rows (`junction_open` guard; neighbour solidity is
itself a pure function of the row, so `row_features` stays pure). On every
already-clean row the guard is a no-op — any subset of a clean placement is
clean — host-proven: rows 0–63 AND the whole replay reach (rows 0–130) are
cell-for-cell identical pre/post fix; the guard only drops one crystal cell
per gate row (and on 9 rows/period the shard then re-hugs the freed edge
cell — all beyond row 2100, far outside any replay's reach).

already exists everywhere (crystals — and v1.1 made them denser with

already there. Score multipliers were rejected (depth-as-score makes a
multiplier muddy); shard-in-danger placement was rejected (needs world
