## 💡 Session idea

The emulator IS a spec disagreement detector — if you refuse to trust
the library. libnds's cardRead/WriteEeprom are the "sanctioned" NDS
save calls, and they are broken on the arc's own proof carrier in a way
no functional test of the happy path would ever show: libnds "deselects"
the backup chip by writing 0x0040 to REG_AUXSPICNT (CARD_SPI_HOLD still
set), real chips shrug, but DeSmuME implements the PROTOCOL — chip
select falls when bit 6 falls, and the command state machine resets on
that edge — so after a libnds-style read every later command byte is
swallowed as more read traffic: the write lands NOWHERE while the ROM's
own counters say it fired, and libnds's UNBOUNDED write-in-progress
poll then reads 0xFF forever and freezes the game solid. The chain that
found and fixed it in one session: (1) a stuck telemetry frame counter
localized the hang to one call; (2) objdump of the shipped libnds9.a
found the unbounded poll; (3) reading the emulator's OWN source (mc.cpp
in the py-desmume sdist — fetched from PyPI when the repo host was
unreachable) explained both symptoms and dictated the correct flow
(drop HOLD before each command's LAST byte); (4) the replacement is 40
lines that speak the same SPI commands with every wait bounded by a
one-constant tunable — sized so an expired poll costs a sub-frame stall
(65536 iterations measurably slipped 2 emulated frames and would have
re-pinned every post-dawn value; 4096 fits inside the dawn-card frame
AND covers a real chip's ~5 ms page program). Lesson for the lane: when
a library call touches hardware your proofs emulate, the emulator's
source is part of the spec — and every poll a game ships should have a
bound and a story for what an expired bound means.

  byte (protocol-correct on hardware, resets DeSmuME's machine).
